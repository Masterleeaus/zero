# TitanZero — Route Name Collisions

> Static analysis of all `->name()` calls across every route file.  
> Only **fully-qualified** route names (prefix chain resolved) are compared.  
> Local name fragments like `index`, `delete`, `list` that appear multiple times  
> are **not** collisions when they resolve to different full names.

---

## Classification Key

| Symbol | Classification | Meaning |
|--------|---------------|---------|
| 🔴 | **Dangerous duplicate** | Same full name, different handler — last registered wins silently |
| 🟠 | **Shadowed handler** | Same full name registered twice in the same file; second wins |
| 🟡 | **Panel vs API duplicate** | Same logical resource with different namespaces — `dashboard.*` vs `api.*` |
| 🟢 | **Safe duplicate** | Same full name, same handler (true idempotent redundancy) |
| 🔵 | **Namespace collision** | Same name fragment in two prefix trees — not a runtime error but confusing |

## Confirmed Collisions

---

### COLL-01 — `webhooks.stripe.success` (🔠 Dangerous duplicate)

**Classification:** 🔴 Dangerous duplicate  
**File:** `routes/webhooks.php` (lines 11 and 14)

| Registration | URI | Handler | Name |
|-------------|-----|---------|------|
| First | `webhooks/stripe/{subscription}/success` | `PaymentProcessController@stripeSuccess` | `webhooks.stripe.success` |
| Second (shadows) | `webhooks/stripe/{plan}/{user}/success/prepaid` | `PaymentProcessController@prepaidStripeSuccess` | `webhooks.stripe.success` |

**Effect:** `route('webhooks.stripe.success')` resolves to the prepaid handler (`prepaidStripeSuccess`) because it is registered second. Any code generating a subscription success URL using the route helper will silently generate a prepaid URL instead.

**Recommended Resolution:**  
Rename one of the two routes. Suggested names:
- `webhooks.stripe.subscription.success` for `stripeSuccess`
- `webhooks.stripe.prepaid.success` for `prepaidStripeSuccess`

```php
// Before
->name('stripe.success')   // subscription
->name('stripe.success')   // prepaid

// After
->name('stripe.subscription.success')
->name('stripe.prepaid.success')
```

### COLL-02 — `webhooks.stripe.cancel` (🔠 Dangerous duplicate)

**Classification:** 🔴 Dangerous duplicate  
**File:** `routes/webhooks.php` (lines 12 and 15)

| Registration | URI | Handler | Name |
|-------------|-----|---------|------|
| First | `webhooks/stripe/{subscription}/cancel` | `PaymentProcessController@stripeCancel` | `webhooks.stripe.cancel` |
| Second (shadows) | `webhooks/stripe/cancel/prepaid` | `PaymentProcessController@stripeCancel` | `webhooks.stripe.cancel` |

**Effect:** Both use the same handler (`stripeCancel`) so there is **no runtime handler error**. The URL generated by `route('webhooks.stripe.cancel')` will point to the prepaid cancel URI. For subscription cancels, any template or redirect using this route name will produce the wrong URL.

**Recommended Resolution:**  
Same pattern as COLL-01:
- `webhooks.stripe.subscription.cancel`
- `webhooks.stripe.prepaid.cancel`

### COLL-03 — `social-media.oauth.webhook.facebook` (🔠 Dangerous duplicate)

**Classification:** 🔴 Dangerous duplicate  
**File:** `routes/core/social.routes.php` (two entries)

| Registration | URI | Handler | Name |
|-------------|-----|---------|------|
| First | `social-media/webhook/instagram` | `SocialInstagramController@webhook` | `social-media.oauth.webhook.facebook` |
| Second (shadows) | `social-media/webhook/facebook` | `FacebookController@webhook` | `social-media.oauth.webhook.facebook` |

**Effect:** The Instagram webhook route was incorrectly given the same name as the Facebook webhook route. `route('social-media.oauth.webhook.facebook')` resolves to the Facebook URI (second wins). More critically, both controllers are **missing from disk** so both will throw `BindingResolutionException` regardless.

**Recommended Resolution:**  
Rename the Instagram webhook to use its own name:

```php
// Before (Instagram webhook — wrong name)
->name('social-media.oauth.webhook.facebook')

// After
->name('social-media.oauth.webhook.instagram')
```

### COLL-04 — `verify-otp` GET and POST (🟠 Shadowed handler)

**Classification:** 🟠 Shadowed handler  
**File:** `routes/auth.php` (lines for GET and POST `verify-otp`)

| Registration | Method | Handler | Name |
|-------------|--------|---------|------|
| First | GET | `AuthenticatedSessionController@verifyOtpCode` | `verify-otp` |
| Second (shadows) | POST | `AuthenticatedSessionController@verifyOtp` | `verify-otp` |

**Effect:** `route('verify-otp')` resolves to the POST route URI (`verify-otp`) because POST is registered second. In practice both point to the same URI, so URL generation is unaffected. However, having two routes with the same name is semantically incorrect and will confuse Laravel's `route()` name lookup.

**Recommended Resolution:**  
Give the GET route its own name:

```php
Route::get('verify-otp', ...)->name('verify-otp.show');
Route::post('verify-otp', ...)->name('verify-otp');
```

### COLL-05 — `dashboard.admin.settings.fal-ai` (🔴 Dangerous duplicate + missing handler)

**Classification:** 🔴 Dangerous duplicate  
**Files:** `routes/panel.php` and `routes/core/social.routes.php`

| Registration | File | Handler | Full Name |
|-------------|------|---------|-----------|
| First (GET) | `panel.php` | `Common\Settings\FalAISettingController@index` | `dashboard.admin.settings.fal-ai` |
| Second (GET, shadows) | `social.routes.php` | `Extensions\SocialMedia\...\FalAISettingController@index` | `dashboard.admin.settings.fal-ai` |
| First (POST) | `panel.php` | `Common\Settings\FalAISettingController@update` | `dashboard.admin.settings.fal-ai.update` |
| Second (POST, shadows) | `social.routes.php` | `Extensions\SocialMedia\...\FalAISettingController@update` | `dashboard.admin.settings.fal-ai.update` |

**Effect:** `social.routes.php` is loaded after `panel.php` in the boot sequence. The social extension definitions shadow the panel definitions for name lookups. Because the social extension controller (`Extensions\SocialMedia\...\FalAISettingController`) is **not present on disk**, navigating to `dashboard/admin/settings/fal-ai` will throw a `BindingResolutionException` rather than render the panel's FAL AI settings page.

**Recommended Resolution:**  
Remove the duplicate `fal-ai` routes from `social.routes.php` since a canonical implementation already exists in `panel.php`. If the social extension requires a different FAL AI handler, use a sub-prefix or a distinct name:

```php
// Option A: Remove from social.routes.php (recommended — panel.php owns this)
// Delete the FalAISettingController block from social.routes.php

// Option B: Give the social extension its own distinct name
->name('dashboard.admin.social-media.settings.fal-ai')
```

### COLL-06 — `dashboard.admin.marketplace.cart` (🟠 Shadowed handler)

**Classification:** 🟠 Shadowed handler  
**File:** `routes/panel.php` (two GET entries for `cart` within the `marketplace.` group)

| Registration | URI | Handler | Name |
|-------------|-----|---------|------|
| First | `dashboard/admin/marketplace/cart` | `MarketPlaceController@cart` | `dashboard.admin.marketplace.cart` |
| Second (shadows, identical) | `dashboard/admin/marketplace/cart` | `MarketPlaceController@cart` | `dashboard.admin.marketplace.cart` |

**Effect:** The route is registered twice with the same URI, handler, and name. This is a harmless no-op at runtime but wastes a route registration and indicates a copy-paste leftover.

**Recommended Resolution:**  
Delete the second `Route::get('cart', ...)` definition in `panel.php` (the duplicate copy).

### COLL-07 — `cache.clear` in `web.php` vs `dashboard.admin.health.cache.clear` in `panel.php`

**Classification:** 🔵 Namespace collision (names differ — not a runtime conflict)

| Registration | File | Full Name | URI |
|-------------|------|-----------|-----|
| web.php | `routes/web.php` | `cache.clear` | `cache-clear` |
| panel.php | `routes/panel.php` | `dashboard.admin.health.cache.clear` | `dashboard/admin/health/cache-clear` |

**Effect:** These are **different** fully-qualified names and do not collide at runtime. However, the fragment `.cache.clear` appearing in a nested name may cause confusion for developers searching by name.

**Recommended Resolution:** No code change required. Consider documenting that `cache.clear` (global, unauthenticated) is distinct from the admin health panel's cache clear action.

### COLL-08 — `upload.image` in `panel.php` vs `dashboard.user.social-media.upload.image` in `social.routes.php`

**Classification:** 🔵 Namespace collision (names differ — not a runtime conflict)

| Registration | File | Full Name |
|-------------|------|-----------|
| `panel.php` | `routes/panel.php` | `upload.image` |
| `social.routes.php` | `routes/core/social.routes.php` | `dashboard.user.social-media.upload.image` |

**Effect:** Different full names — no runtime collision. The short `upload.image` name in panel.php is a global utility route without a dashboard prefix, which is unusual compared to the rest of the system.

**Recommended Resolution:** Consider prefixing the panel utility route under `dashboard.` or a utility namespace for consistency. Low priority.

### COLL-09 — Panel vs API TitanRewind routes (🟡 Panel vs API duplicate)

**Classification:** 🟡 Panel vs API duplicate  
**Files:** `routes/core/rewind.routes.php` and `routes/api.php`

| Context | Full Name Prefix | Controller | Purpose |
|---------|-----------------|-----------|---------|
| Dashboard (web) | `titanrewind.*` | `TitanRewindCaseController` | Web UI |
| API | `api.titanrewind.*` | `TitanRewindApiController` | Mobile/API client |

**Effect:** No runtime name collision — the prefixes are different (`titanrewind.` vs `api.titanrewind.`). This is intentional separation between web and API surfaces.

**Note:** The `titanrewind.` prefix in `rewind.routes.php` deviates from the dominant `dashboard.user.*` convention used by most other user-facing routes. See `ROUTE_MANIFEST_FULL.md §4` for full deviation list.

**Recommended Resolution:** Consider renaming `rewind.routes.php` prefix from `titanrewind.` to `dashboard.user.titanrewind.` to align with platform naming conventions. This would require updating all `route('titanrewind.*')` references in views and controllers.

3. **COLL-03** — Instagram webhook has wrong name; any code generating webhook URLs for Instagram will produce the Facebook webhook URL.

6. **COLL-04** — `verify-otp` both GET and POST share the same name; functional but semantically wrong.
7. **COLL-07, COLL-08, COLL-09** — Informational only; no code changes required for correctness.
